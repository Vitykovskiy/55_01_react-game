В ветках для разработки допускается свободное написание коммитов. Но, когда над задачей в одной ветке работает несколько разработчиков, желательно учитывать лучшие практики.

### **0. Именование ветки.**

Название должно быть осмысленным по отношению, что делается в ветке
Именуем feature ветки: `CODE-Feature-branch-name`
Аналогично с fix: `CODE-Fix-branch-name`

`CODE` - порядковый номер issue, над которым работаете

### **1. Коммит/PR должен быть атомарным, добавляющим целостное, не ломающее изменение кода.**

### **2. PR/Коммиты из feature веток должны попадать в общие ветки всегда одним коммитом, например, с помощью --squash, без использования merge коммита.**

Разработку ведем в своей ветке. После завершения разработки все промежуточные коммиты, которые были сделаны в процессе разработки, сквошим в единый коммит.

И только единый коммит должен попасть в общую ветку.

### **3. При написании сообщений коммитов придерживаемся стандарта написания коммитов [Conventional Commits](https://www.conventionalcommits.org/).**

Стандарт предполагает следующую структуру PR/коммита:

```
[optional Pull Request #<pr_number>] <type>[optional scope]: <description>

[optional body]

[optional footer]
```

Пример:

```
Pull Request 27 Добавлена страница регистрации

Добавлены вёрстка, роут, новый компонент FormFieldPassword
```

- У коммита обязательно должен быть заголовок, а также необязательно может быть тело и футер.
- Для детализации коммита используем тело коммита. Тело должно быть отделено от заголовка пустой строкой. Тело коммита необязательный элемент.
- Футер следует использовать для указания внешних ссылок, контекста коммита или другой мета информации. Футер коммита необязательный элемент.
- Футер отделяется от тела сообщения пустой строкой.

### **4. При написании сообщений коммитов придерживаемся следующих практик**

- Заголовок коммита должен отвечать на вопросы "Что" изменилось и "Почему".
- Заголовок коммита должен начинаться с большой буквы.
- В конце заголовка коммита не должно быть знаков препинания.
- Заголовок коммита не должен превышать ~80 символов. В общем случае придерживаемся того, чтобы заголовок при просмотре истории коммитов не выходил за пределы экрана.
- Сообщение должно быть полезным. Не нужно перечислять произведенные изменения, которые видны в diff.
- Сообщение не должно содержать "воды", мусорных слов и эмоций. Оно должно быть максимально точным и конкретным.
- Тело комментария должно быть достаточно подробным. И также, как и заголовок, тело комментария должно отвечать на вопросы "Что" изменилось и "Почему", но в более развернутой форме.
- Сообщение коммита должно содержать глаголы.
- Глаголы пишем в прошедшем времени от третьего лица (исправлено, добавлено и т.д.)
