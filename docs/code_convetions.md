1. **Простота и читаемость**

   - Следуем принципам **KISS** (Keep It Simple, Stupid), **DRY** (Don't Repeat Yourself), **YAGNI** (You Aren’t Gonna Need It) и т.д.
   - Не пишем сложные конструкции ради «умности», код должен быть понятен другим разработчикам.

2. **Функции и методы**

   - Функция не должна превышать один экран. Если становится больше — делим на несколько с понятными именами.
   - Даже если функция не переиспользуется, ясность важнее краткости.
   - Имена функций отражают действие: `calculateTotal`, `fetchUserData`, `renderCard`.

3. **Именование переменных и структур**

   - Имена переменных должны быть **смысловыми**, избегаем `a`, `b`, `data1`.
   - Константы пишем **в верхнем регистре** с подчеркиваниями: `MAX_ATTEMPTS`.
   - Объекты и классы — с `PascalCase`, функции и методы — `camelCase`.

4. **Комментарии**

   - Поясняем **почему**, а не **что** код делает.
   - Не пишем комментарии, которые дублируют код.
   - TODO/FIXME оформляем явно, чтобы можно было искать:

     ```ts
     // TODO: добавить валидацию email
     ```

5. **Обработка ошибок**

   - Всегда обрабатываем возможные исключения или ошибки.
   - Не оставляем «тихие» try/catch без логирования(логирование в console).
   - Используем понятные сообщения для логов и ошибок.

6. **Типизация и структуры данных**

   - В TypeScript/JS всегда стараемся указывать типы там, где это повышает читаемость и безопасность.
   - Используем интерфейсы и типы для сложных структур:

     ```ts
     interface User {
       id: number
       name: string
       email: string
     }
     ```

7. **Работа с зависимостями**

   - Стараться минимизировать внешние библиотеки.
   - Всегда фиксируем версии пакетов, чтобы сборка была стабильной.
   - Обновления зависимостей делаем через PR с проверкой, что ничего не сломалось.

8. **Рефакторинг и улучшения**

   - Не боимся рефакторить код, если можно улучшить читаемость или производительность.
   - Всегда проверяем, что изменения не ломают существующую функциональность.

9. **Базовые компоненты должны быть качественными и расширяемыми**

   - Если компонент (UI или функция) используется во многих местах приложения, пишем его с учётом будущих изменений.
   - Компонент должен быть расширяемым, а не требовать изменений в коде при добавлении нового функционала.
   - Предпочтение отдаём композиции и конфигурируемым параметрам вместо прямого изменения внутренней логики.
