**Архитектура FSD**
1. Слои (Layers) — «Этажи нашего здания»
Это самое главное правило — вертикальное разделение ответственности. Представь многоэтажное здание:
   Shared (Общий) — Фундамент и коммуникации. Здесь лежит то, что используется повсюду: базовые компоненты (кнопки, поля ввода), утилиты (функции для работы с датами, строками), хранение состояния. Всё, что не относится к бизнесу.
   Entities (Сущности) — Несущие стены и перекрытия. Это ключевые бизнес-понятия вашего приложения: User (Пользователь), Product (Товар), Order (Заказ). Они не знают о том, что выше, и опираются на общий слой.
   Features (Фичи) — Квартиры с готовым ремонтом. Это независимые, полезные для пользователя возможности, собранные из сущностей. Например, «Добавить товар в корзину», «Аутентифицировать пользователя», «Поиск по каталогу». Их можно переиспользовать.
   Widgets (Виджеты) — Комнаты. Это более крупные блоки, которые собираются из фич и сущностей. Например, «Сайдбар с навигацией», «Шапка профиля», «Карточка товара».
   Pages (Страницы) — Готовые этажи. Это полноценные страницы приложения (например, «Главная», «Страница товара», «Корзина»). Они компонируют виджеты и фичи в нужном порядке.
   App (Приложение) — Крыша и подъезд. Самый верхний уровень, который отвечает за сборку всего приложения: настройка роутинга (навигации между страницами), провайдеров (глобальные темы, стили) и запуск.
Главный принцип: Верхние слои могут «звать» или использовать нижние, но не наоборот. Страница (Pages) может использовать виджет (Widget), но виджет не может напрямую обращаться к странице. Это обеспечивает стабильность и предсказуемость.

2. Слайсы (Slices) — «Квартиры на одном этаже»
Внутри каждого этажа (слоя) есть много квартир. Это и есть слайсы — горизонтальное разделение по бизнес-доменам.
На слое Entities у тебя будут слайсы (квартиры) user, product, order.
На слое Features — слайсы addToCart, userProfile, productSearch.
Важное правило соседства: Квартиры на одном этаже не могут иметь общих дверей. То есть, слайсы на одном слое не могут импортировать друг друга. Это предотвращает запутывание и создание зависимостей между несвязанными частями приложения.

3. Сегменты (Segments) — «Планировка внутри квартиры»
В каждой квартире (слайсе) есть комнаты разного назначения. Это техническое разделение кода внутри одного слайса.
   ui — Внешний вид и интерфейс. Здесь лежат компоненты, стили, всё, что видит пользователь.
   model — Логика и данные. Бизнес-логика, хранилища состояний (например, Redux сторы), типы данных.
   api — Общение с сервером. Запросы к бэкенду.
   lib — Вспомогательные инструменты. Хелперы, утилиты, специфичные для этого слайса.
   config — Настройки. Конфигурационные файлы.
Это разделение помогает быстро найти нужный код. Хочешь изменить внешний вид — иди в ui, меняешь логику — открывай model.

4. Правила импортов — «Кто кого может звать в гости»
   Это прямое следствие из структуры «здания»:
   Разрешено импортировать только сверху вниз.
   App → Pages → Widgets → Features → Entities → Shared
Страница может использовать виджет, фичу или сущность.
Фича может использовать другую фичу (только если она находится на том же или более низком слое!), сущность или общий код.
Сущность не может использовать фичу или виджет! Она фундаментальна и не должна знать о том, как её используют.

5. Публичный API — «Входная дверь в квартиру»
Представь, что у каждой квартиры (слайса) есть только одна входная дверь — файл index.ts. Если ты хочешь что-то отдать соседям (другим частям приложения), ты кладешь это у себя у входа.
Внутри своей квартиры ты можешь делать ремонт как угодно: переставлять мебель (ui), менять проводку (model).
Но если ты хочешь, чтобы друг из другой квартиры мог воспользоваться твоим диваном (компонентом) или инструментом (функцией), ты выносишь его к входной двери (экспортируешь в index.ts).
Это правило максимально важно. Оно скрывает внутреннюю сложность каждого модуля. Снаружи все видят только красивую, понятную дверь, а не весь бардак внутри (если он есть). Это делает код надежным и легко рефакторимым.

6. Пример:
   src/
   ├── app/                    # Инициализация приложения
   │   ├── routes/            # Маршрутизация
   │   ├── styles/            # Глобальные стили
   │   ├── App.tsx            # Собирает ВСЕ слои вместе
   │   └── index.tsx          # Точка входа 
   │
   ├── pages/                 # Страницы
   │   ├── LoginPage          # Пример страницы
   │       ├── model          # Модель страницы
   │       ├── ui             # Компоненты страницы
   │       ├── api            # API слой страницы
   │       └── index.tsx      # Public API
   │   └── index.tsx          # Реэкспорт
   │
   ├── shared/                # Общие модули
   │   ├── ui/                # Кнопки, инпуты - НЕ знают о бизнесе
   │   ├── lib/               # HTTP-клиент, утилиты
   │   └── config/            # Настройки
